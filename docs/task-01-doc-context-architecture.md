# Task 01 Doc. Контекст проекта и базовая архитектура

## Краткое введение

DNS, Domain Name System, нужен для того, чтобы приложения и люди обращались к
сервисам по именам, а не запоминали IP-адреса. Например, в этой лаборатории имя
`web.internal` переводится в адрес `10.10.0.20`.

Проект показывает не только один DNS-запрос, а небольшую инфраструктуру:
авторитативные зоны, рекурсию, master/slave, безопасность, разные ответы для
разных клиентов и автоматическую проверку результата.

## Почему выбран такой способ

Для проекта выбран BIND9, потому что он умеет быть и авторитативным DNS-сервером
для локального домена, и рекурсивным резолвером для доверенной сети. Это
позволяет реализовать все требования задания в одном понятном стеке.

Docker Compose создает одинаковую лабораторию на любой машине с Docker:
контейнеры получают фиксированные адреса, а конфигурация и зоны хранятся в git.
Python-тесты дополняют ручные команды `dig` и доказывают, что ответы
воспроизводимы.

## Основы DNS

Домен - это имя в иерархии DNS. В проекте используется локальный домен
`internal`, а имена сервисов имеют вид `web.internal` или `mail.internal`.

Зона - это административная часть пространства имен, за которую отвечает
конкретный DNS-сервер. Файл зоны `internal` содержит записи локальных сервисов.
Reverse zone `0.10.10.in-addr.arpa` содержит обратные соответствия для адресов
сети `10.10.0.0/24`.

Основные типы записей:

| Тип | Назначение | Пример в проекте |
| --- | --- | --- |
| `A` | Имя в IPv4-адрес | `web.internal -> 10.10.0.20` |
| `AAAA` | Имя в IPv6-адрес | `web.internal -> fd00:10:10::20` |
| `MX` | Почтовый сервер домена | `internal -> mail.internal` |
| `TXT` | Текстовые метаданные | `course=dns-project` |
| `CNAME` | Псевдоним другого имени | `www.internal -> web.internal` |
| `PTR` | IP-адрес в имя | `10.10.0.20 -> web.internal` |

Forward DNS отвечает на вопрос "какой адрес у имени?". Reverse DNS отвечает на
вопрос "какое имя связано с адресом?". Reverse DNS полезен для диагностики,
логов и систем, которые проверяют соответствие адреса имени.

## Авторитативный и рекурсивный DNS

Авторитативный DNS-сервер хранит зону и дает окончательный ответ по ее именам.
`dns-master` и `dns-slave` авторитативны для внутренней зоны `internal`.

Рекурсивный DNS-сервер получает запрос клиента и при необходимости ищет ответ у
других DNS-серверов. В проекте `dns-master` выполняет рекурсию только для
клиентов доверенной сети `10.10.0.0/24`. Недоверенная внешняя сеть не может
использовать его как открытый резолвер.

Обе роли нужны, потому что локальные имена должны обслуживаться авторитативно, а
доверенные клиенты должны уметь разрешать внешние имена через тот же сервер.

## Master и slave

Master DNS хранит исходные редактируемые файлы зон. Slave DNS получает копии зон
с master через zone transfer и может продолжать отвечать клиентам, если master
временно недоступен.

В проекте transfer разрешен только адресу slave-сервера `10.10.0.3`. Обычный
клиент не может скачать всю зону через AXFR.

## Реализованная архитектура

```text
trusted network: dns-lab, 10.10.0.0/24

dns-client (10.10.0.10)
        |
        | dig / pytest / recursion
        v
dns-master (10.10.0.2) -------- AXFR/IXFR -------> dns-slave (10.10.0.3)
        |
        | DNSSEC-validating recursion
        v
upstream DNS

external network: dns-external, 10.20.0.0/24

dns-external-client (10.20.0.10) ---> dns-master (10.20.0.2)
                                      external view, no recursion
```

| Компонент | Назначение |
| --- | --- |
| `dns-master` | Исходные зоны, рекурсия, DNSSEC validation, views, logging, `rndc`, RRL |
| `dns-slave` | Копии внутренних зон и авторитативные ответы |
| `dns-client` | Доверенные ручные и автоматические проверки |
| `dns-external-client` | Проверка ACL и внешнего Split-Horizon ответа |

## BIND9 и Unbound

Сравнивать DNS-серверы важно, потому что они могут быть оптимизированы под
разные роли.

BIND9 является полноценным авторитативным и рекурсивным DNS-сервером. В этом
задании к BIND9 относятся forward/reverse зоны, master/slave, AXFR, ACL,
DNSSEC validation, `logging {}`, `rndc`, Split-Horizon через `view` и RRL.

Unbound прежде всего известен как быстрый и безопасный рекурсивный кэширующий
резолвер. Он мог бы выполнять рекурсию, DNSSEC validation и ограничение доступа
клиентов, но не является удобной заменой BIND9 для учебной master/slave
авторитативной инфраструктуры.

BIND9 и Unbound можно использовать одновременно в более сложной архитектуре:
BIND9 обслуживает авторитативные зоны, а Unbound отдельно принимает
рекурсивные запросы. В этом проекте Unbound не реализуется, потому что BIND9
уже закрывает требования задания без лишнего усложнения.

## Принятые значения

| Параметр | Значение |
| --- | --- |
| Домен | `internal` |
| Доверенная сеть | `10.10.0.0/24` |
| Внешняя демонстрационная сеть | `10.20.0.0/24` |
| Master DNS | `10.10.0.2` |
| Slave DNS | `10.10.0.3` |
| Reverse zone | `0.10.10.in-addr.arpa` |
| Внутренний web-адрес | `10.10.0.20` |
| Внешний демонстрационный web-адрес | `203.0.113.20` |

## Разделение работы

Один участник может отвечать за Docker Compose, master DNS, зоны, logging и
`rndc`. Второй участник может отвечать за slave, AXFR, ACL, DNSSEC,
Split-Horizon, RRL, тесты и документацию. Такое разделение сохраняет понятные
границы, но итог проверяется как единая инфраструктура.
